Structuring data for dTree is hard... but not anymore! Painlessly, with just one method call, you can:

  • Filter data for the following objects:
    • Target (the person whose family tree is being displayed)
    • Target's parents (parents of target)
    • Target's siblings (that share both parents)
    • Target's children (where target is listed a parent)
    • Target's spouses (where listed as other parent of a child)
    • Target's descendents (grandchildren, children's spouses, great-grandchildren, etc.)
  • Dynamically set CSS classes and custom render data for each node.
  • Return a hierarchical data structure, formatted as specified by dTree's README.

dTree-Seed is a JavaScript library that converts a list of family members into a hierarchical family tree data structure. It serves as a pre-processor for dTree, a library for displaying family trees. 

                
// just one method call
seededData = dSeeder.seed(data, targetId);
// then pass to dTree
dTree.init(seededData);

I used dTree when building PicsOfUs, and found it to be a great tool that saved me a lot of time trying to figure out how to display family trees. Within PicsOfUs resides the logic for filtering a list of objects for related family members and converting that into a hierarchical data structure.

After I finished coding PicsOfUs, I saw an opportunity to extend a library that helped me and give back to the open-source community. My goal was to condense the filtering and structuring logic from PicsOfUs into a package allowing someone to get started with dTree quickly and make it more dynamic overall.

I wanted to use the same language as dTree, Javascript. Still, I tried to keep the development time going smoothly, avoiding run-time errors, so I took a step back for a few weeks and learned Typescript.

There weren't many issues because I started from the beginning with test-driven-development. I created MockData classes for tests, and things progressed nicely.

With all the logic finished, it was time for me to write the README; explanations and documentation came easier since I learned a few things after designing the Work With Me Here tutorial. Although there was one major issue I hadn't considered, the (external) module pattern I used required the user to set up the few files I had together. It would take a lot of work to figure out for a beginner. I mean, I couldn't figure it out!

I converted the dTree-Seed to use the namespace (internal) pattern. I changed the tsconfig.json to transpile all the code into one file. Now, dTree-Seed would be available in a global variable and all the logic would be behind one method, just like dTree, perfect!

Something I found surprising was there aren't any testing frameworks compatible with the namespace Typescript pattern. I think this has to do with the community's (understandable) aversion to global variables and the popularity of the module pattern. Most testing frameworks, especially those compatible with Typescript, are created for Node.js, for which the module pattern is recommended.

I'm happy with the outcome of dTree-Seed. I'm pleased with the outcome. dTree-Seed works very well with dTree; all you have to do is call one method to arrange your data.

I learned a lot about Typescript and testing, and I'm glad I was able to give back to the open-source community. I hope dTree-Seed helps someone else save time and get started with dTree. If you have any questions or comments, feel free to reach out to me. I'd love to hear from you!
