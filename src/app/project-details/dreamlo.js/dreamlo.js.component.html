• Create, retrieve, update, and delete scores from a specified leaderboard via HTTP requests to dreamlo's API.
  • Return scores in various sort orders and formats, including Javascript objects, JSON, XML, pipe-delimited, and quoted with comma separators.
  • Optionally skip over scores to serve as in separate paginated sections, saving memory.
  • Ships with dreamlo UI, a simple interface allows you to test/debug your leaderboard without writing any code.

dreamlo is a website for adding leaderboards to games for free. I've previously used its Unity game asset for Work With Me Here. Later, I revisited an old TicTacToe game I made while learning how to code, to see how my coding style has changed over time; then, I thought, "What would be a good way to add a leaderboard to these types of games?" The first thing that came to mind was dreamlo.

My goal was to create an easy-to-integrate script containing all the functionality dreamlo offers and host it as open-source so others may also take advantage of this free tool. An HTML game is often one of a beginner's first programs; I wanted dreamlo.js to be as easy to use as possible for someone to extend their game because I know the excitement of creating something.

Can you climb to the top?

Using dreamlo.js, I added a leaderboard to that old tic-tac-toe project of mine for consecutive wins against the CPU. Can you beat my high score? Play Now

In the beginning, it seemed straightforward; all I had to do was make a Javascript wrapper for dreamlo's requests, but I ended up with more than just that.

Implementing the basic methods was simple because I had dreamlo's developer page to guide me. However, during testing, I discovered dreamlo itself had bugs that I needed to compensate for:

  • Retrieving a single score using a name, if no scores had that name, dreamlo would return all scores instead of an empty result. This issue only affected the JSON, XML, and Quoted with comma formats; pipe-delimited scores seemed to function normally.
  • Retrieving multiple scores using the JSON format, the returned data would be different types when there were no scores, one or numerous scores. I had to implement a fix to enforce data was always in an array to keep it consistent.

Since this script is in Javascript, I decided to extend the number of formats by one to include an option to have scores returned as objects. I manipulated the JSON format to make this possible.

After I finished the script, I had another monster to contend with, writing the documentation. I broke up the information in a digestible way, providing relevant links to outside documentation if the reader isn't familiar with a concept. In the end, it took 45 commits to get the README to a level I was satisfied with.

I spent two weeks working on a Swagger UI-inspired demo page for debugging purposes. I wanted someone to check out dreamlo.js before integrating it or debugging their leaderboard to see if it, or their project, was the cause of an issue.

I decided to use Typescript for this project. Its statically-typed nature is more familiar to me, having written primarily C# in the past. Ultimately, it saved me time, unlike if I used dynamically-typed JavaScript, where I would only realize type errors at run-time.

For dreamlo UI, I used Bootstrap to get me off the ground in terms of design quickly. Where necessary, I used CSS classes that mimicked Bootstrap to customize styles to achieve a look that's a cross between Swagger UI and dreamlo's official site design.

I'm very proud of dreamlo.js. I'm happy that I could contribute to the open-source community, which has helped me with so many free libraries in my projects. I got much more comfortable with Typescript, which was great practice. I prefer Typescript over vanilla Javascript now!

Throughout this project, my source code commits became much cleaner and more descriptive, which helped me track down changes much faster.
